Design note: sharing the prd PAPI descriptors

Status: design proposal - not implemented. Scope: how to publish the generated polkadot-api (PAPI) prd descriptor set plus the committed chain metadata as one shared package, co-located with @predictor-foundation/papi-chain, so every consumer stops regenerating its own copy.

1. How it works today

The faucet backend is the only current PAPI consumer. Its setup:

  • Committed metadata. faucet/packages/backend/.papi/metadata/prd.scale is checked in. .papi/polkadot-api.json pins the entry: 
    {
  "version": 0,
  "descriptorPath": ".papi/descriptors",
  "entries": {
    "prd": {
      "wsUrl": "wss://chain-external.testnet.prdctr.io",
      "metadata": ".papi/metadata/prd.scale",
      "genesis": "0x082967ba30c9cdd763afaf0dfcb573f6ce89233cfe1e22116f0d1a672db93980",
      "codeHash": "0xa3e6150b8faa189f8dec4647483ecffd80a3cc35c8b300ccac80c84829937084"
    }
  }
}
  • Generated descriptors. papi compiles the metadata into a local package at .papi/descriptors, referenced as "@polkadot-api/descriptors": "file:.papi/descriptors". The generated dist/ is gitignored; only package.json, generated.json, and the metadata are tracked.
  • Regeneration hook. "postinstall": "papi" rebuilds the descriptors after every install. In Docker the faucet backend image re-runs pnpm exec papi explicitly (offline, from the committed prd.scale) because the gitignored dist/ is absent from the build context.
  • Consumption. chain.ts imports { prd } from @polkadot-api/descriptors and injects it into the descriptor-generic Chain<typeof prd> from @predictor-foundation/papi-chain. papi-chain deliberately ships no descriptors and is pinned to no runtime version - each consumer supplies its own D.

Why this is a problem when a second consumer appears

  • Metadata duplication / drift. Each new PAPI consumer would check in its own prd.scale. The moment the runtime upgrades, they drift, and two services build against two different chain shapes with no single place to bump.
  • Per-consumer toolchain coupling. Every consumer needs the papi CLI, a postinstall hook, the .papi/ layout, and Docker-time regeneration logic. That is a lot of identical scaffolding, and the postinstall hook makes CI installs slower and network/tooling-sensitive.
  • The endpoint constant is duplicated. wsUrl in polkadot-api.json repeats the same wss://chain-external.testnet.prdctr.io that already lives in the faucet's chain.ts and now in @predictor-foundation/network-constants.

2. Options

Option 1 - Ship a prebuilt descriptors package (recommended)

Add a new workspace package, @predictor-foundation/papi-descriptors, next to papi-chain. It owns the committed prd.scale, runs papi at its own build time, and publishes the already-generated descriptor dist/ as normal package output (like every other lib/ in the monorepo). Consumers depend on it and import { prd } from "@predictor-foundation/papi-descriptors" - no papi CLI, no postinstall, no .papi/ folder, no Docker regeneration step.

  • Pros: one source of truth for metadata; consumers are plain library imports; Docker images lose the offline-regeneration dance; version bumps are a single PR + semver release; pairs naturally with papi-chain's descriptor-generic design (Chain<typeof prd>).
  • Cons: the generated dist/ must be committed or reliably produced in CI before publish (see build-hook note below); the generated code can be large; the package must publish real files rather than the current gitignored layout.

Option 2 - Commit the generated descriptors in-repo, drop postinstall

Keep the descriptors inside each consumer's .papi/, but check the generated dist/ into git and remove the postinstall hook and Docker regeneration. Installs and image builds get simpler.

  • Pros: smallest change; no new package.
  • Cons: does not solve duplication - each consumer still carries metadata + descriptors; drift risk remains; large generated blobs land in every consumer repo.

Option 3 - Fold descriptors into papi-chain

Have papi-chain itself export prd.

  • Pros: one dependency for consumers.
  • Cons: breaks the package's core secret. papi-chain is intentionally descriptor-generic and runtime-agnostic; baking in the testnet's prd descriptors couples the reusable boundary to one chain version and forces a papi-chain release on every runtime upgrade. Rejected - it violates the module's stated boundary.

3. Recommendation

Option 1: a dedicated @predictor-foundation/papi-descriptors package. It keeps papi-chain generic (its secret is the connect/submit plumbing), gives the prd metadata a single owner, and turns every consumer into an ordinary import. Sketch: 

packages/papi-descriptors/
├── package.json        # name @predictor-foundation/papi-descriptors, main lib/index.js
├── .papi/
│   ├── polkadot-api.json   # the single committed prd entry (metadata + genesis + codeHash)
│   └── metadata/prd.scale  # the ONE committed metadata blob for the whole org
├── src/index.ts        # re-export: export { prd } from "@polkadot-api/descriptors"
└── lib/                # tsc + papi output, published (files: ["lib/"])
  • Generation: run papi as a build step (not postinstall), so it fires once when the descriptors package is built, not on every consumer install. Commit the generated dist/ (or generate-then-publish in CI) so downstream installs and Docker builds need no PAPI toolchain at all.
  • Endpoint reuse: the wsUrl in polkadot-api.json should match PREDICTOR_TESTNET.wsEndpoints[0] from @predictor-foundation/network-constants. Keep network-constants as the human-facing source of truth for the endpoint; polkadot-api.json only needs it for the (optional) papi update refresh-from-chain flow.
  • Migration for the faucet: replace its file:.papi/descriptors dependency and postinstall with a dependency on @predictor-foundation/papi-descriptors; delete its .papi/ and the Docker regeneration block (shape A template's optional papi section then drops out entirely).
  • Versioning: semver the descriptors package; a runtime metadata upgrade is a normal minor/major release that consumers pin and bump deliberately - the RAD "version your formats from day one" default for an on-chain-shaped artifact many consumers depend on.

Open question to settle before implementing: commit the generated dist/ vs. generate-in-CI-before-publish. Committing is simplest and makes the package installable straight from git; generating in CI keeps the repo smaller but requires the papi toolchain in the release pipeline. Given the rest of the monorepo commits build output sparingly and relies on a central build, generate-in-CI aligns better - but that decision is out of scope for this note.
